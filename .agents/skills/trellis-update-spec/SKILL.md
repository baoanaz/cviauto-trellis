# 更新 Code-Spec — 沉淀可执行契约

当你学到有价值的内容（来自调试、实现或讨论），使用此技能更新相关的 code-spec 文档。

**时机**：完成任务后、修复 bug 后、或发现新模式后

---

## Code-Spec 优先规则（关键）

在此项目中，用于实现工作的 "spec" 指 **code-spec**：
- 可执行契约（而非仅原则性文本）
- 具体的签名、payload 字段、env key 和边界行为
- 可测试的验证/错误行为

如果变更涉及基础设施或跨层契约，code-spec 深度是强制性的。

### 强制触发器

当变更包括以下任何内容时，应用 code-spec 深度：
- 新增/变更的命令或 API 签名
- 跨层请求/响应契约变更
- 数据库 schema/迁移变更
- 基础设施集成（存储、队列、缓存、密钥、环境变量接线）

### 强制输出（7 个部分）

对于触发的任务，包含以下所有部分：
1. 范围 / 触发器
2. 签名（命令/API/DB）
3. 契约（请求/响应/环境变量）
4. 验证与错误矩阵
5. 好/基本/坏案例
6. 需要的测试（含断言点）
7. 错误 vs 正确（至少一对）

## 何时更新 Code-Spec

| 触发器 | 示例 | 目标 Spec |
|---------|---------|-------------|
| **实现了一个功能** | 添加了新的集成或模块 | 相关 spec 文件 |
| **做了一个设计决策** | 选择了可扩展性模式而非简单模式 | 相关 spec + "设计决策"部分 |
| **修复了一个 bug** | 发现了错误处理中的微妙问题 | 相关 spec（如 error-handling 文档） |
| **发现了一个模式** | 找到了更好的代码结构方式 | 相关 spec 文件 |
| **踩了一个坑** | 了解到 X 必须在 Y 之前做 | 相关 spec + "常见错误"部分 |
| **建立了一个约定** | 团队就命名模式达成一致 | 质量指南 |
| **新的思维触发器** | "别忘了在做 Y 之前检查 X" | `guides/*.md`（作为检查清单项） |

**关键洞察**：Code-spec 更新不仅是为了问题。每个功能实现都包含设计决策和契约，未来的 AI/开发者需要这些才能安全执行。

## Spec 结构概览

```
.trellis/spec/
├── <layer>/           # 每层的编码标准（如 backend/、frontend/、api/）
│   ├── index.md       # 概览和链接
│   └── *.md           # 主题特定指南
└── guides/            # 思维检查清单（不是编码 spec！）
    ├── index.md       # 指南索引
    └── *.md           # 主题特定指南
```

### 关键：Code-Spec vs Guide — 了解区别

| 类型 | 位置 | 目的 | 内容风格 |
|------|----------|---------|---------------|
| **Code-Spec** | `<layer>/*.md` | 告诉 AI"如何安全实现" | 签名、契约、矩阵、案例、测试点 |
| **Guide** | `guides/*.md` | 帮助 AI"需要考虑什么" | 检查清单、问题、指向 spec 的指针 |

**决策规则**：问自己：

- "这是**如何编写**代码" → 放入 spec 层目录
- "这是编写前**需要考虑什么**" → 放入 `guides/`

**示例**：

| 学到的东西 | 错误位置 | 正确位置 |
|----------|----------------|------------------|
| "对此任务使用 API X 而非 API Y" | ❌ `guides/`（对思维指南太具体） | ✅ 相关 spec 文件（具体约定） |
| "做 Y 时记得检查 X" | ❌ Spec 文件（对 spec 太抽象） | ✅ `guides/`（思维检查清单） |

**指南应当是简短的检查清单，指向 spec**，而非重复详细规则。

## 更新流程

### 步骤 1：识别你学到了什么

回答以下问题：

1. **你学到了什么？**（要具体）
2. **为什么重要？**（能防止什么问题？）
3. **它属于哪里？**（哪个 spec 文件？）

### 步骤 2：分类更新类型

| 类型 | 说明 | 行动 |
|------|-------------|--------|
| **设计决策** | 为什么选择方案 X 而非 Y | 添加到"设计决策"部分 |
| **项目约定** | 此项目中如何做 X | 添加到相关部分并附示例 |
| **新模式** | 发现的可复用方法 | 添加到"模式"部分 |
| **禁止模式** | 会引起问题的做法 | 添加到"反模式"或"不要"部分 |
| **常见错误** | 容易犯的错误 | 添加到"常见错误"部分 |
| **约定** | 达成一致的标准 | 添加到相关部分 |
| **陷阱** | 不显然的行为 | 添加警告提示 |

### 步骤 3：阅读目标 Code-Spec

编辑前，阅读当前 code-spec 以：
- 了解现有结构
- 避免重复内容
- 找到正确的更新位置

```bash
cat .trellis/spec/<category>/<file>.md
```

### 步骤 4：进行更新

遵循以下原则：

1. **要具体**：包含具体示例，而非仅仅抽象规则
2. **解释为什么**：说明能防止什么问题
3. **展示契约**：添加签名、payload 字段和错误行为
4. **展示代码**：为关键模式添加代码片段
5. **保持简短**：每部分一个概念

### 步骤 5：更新索引（如需要）

如果添加了新部分或 code-spec 状态发生了变化，更新该类别的 `index.md`。

## 更新模板

### 基础设施/跨层工作的强制模板

```markdown
## 场景：<名称>

### 1. 范围 / 触发器
- 触发器：<为什么需要 code-spec 深度>

### 2. 签名
- 后端命令/API/DB 签名

### 3. 契约
- 请求字段（名称、类型、约束）
- 响应字段（名称、类型、约束）
- 环境变量 key（必需/可选）

### 4. 验证与错误矩阵
- <条件> -> <错误>

### 5. 好/基本/坏案例
- 好：...
- 基本：...
- 坏：...

### 6. 需要的测试
- 单元/集成/E2E 及断言点

### 7. 错误 vs 正确
#### 错误
...
#### 正确
...
```

### 添加设计决策

```markdown
### 设计决策：[决策名称]

**上下文**：我们在解决什么问题？

**考虑的选项**：
1. 选项 A - 简要说明
2. 选项 B - 简要说明

**决策**：我们选择了选项 X，因为...

**示例**：
\`\`\`typescript
// 如何实现
代码示例
\`\`\`

**可扩展性**：未来如何扩展此方案...
```

### 添加项目约定

```markdown
### 约定：[约定名称]

**是什么**：约定的简要描述。

**为什么**：为什么在此项目中这样做。

**示例**：
\`\`\`typescript
// 如何遵循此约定
代码示例
\`\`\`

**相关**：相关约定或 spec 的链接。
```

### 添加新模式

```markdown
### 模式名称

**问题**：此模式解决什么问题？

**解决方案**：方法的简要描述。

**示例**：
\`\`\`
// 好的做法
代码示例

// 不好的做法
代码示例
\`\`\`

**为什么**：解释为什么这种方式更好。
```

### 添加禁止模式

```markdown
### 不要：模式名称

**问题**：
\`\`\`
// 不要这样做
不好的代码示例
\`\`\`

**为什么不好**：问题的解释。

**替代方案**：
\`\`\`
// 改为这样做
好的代码示例
\`\`\`
```

### 添加常见错误

```markdown
### 常见错误：描述

**症状**：出了什么问题

**原因**：为什么会发生

**修复**：如何改正

**预防**：未来如何避免
```

### 添加陷阱

```markdown
> **警告**：对不显然行为的简要描述。
>
> 关于何时发生以及如何处理的详细信息。
```

## 交互模式

如果不确定更新什么，回答以下提示：

1. **你刚完成了什么？**
   - [ ] 修复了一个 bug
   - [ ] 实现了一个功能
   - [ ] 重构了代码
   - [ ] 讨论了方法

2. **你学到了或决定了什么？**
   - 设计决策（为什么选 X 而非 Y）
   - 项目约定（我们如何做 X）
   - 不显然的行为（陷阱）
   - 更好的方法（模式）

3. **未来的 AI/开发者需要知道这个吗？**
   - 为了理解代码如何工作 → 是，更新 spec
   - 为了维护或扩展功能 → 是，更新 spec
   - 为了避免重复错误 → 是，更新 spec
   - 纯粹的一次性实现细节 → 也许跳过

4. **它与哪个领域相关？**
   - [ ] 后端代码
   - [ ] 前端代码
   - [ ] 跨层数据流
   - [ ] 代码组织/复用
   - [ ] 质量/测试

## 质量检查清单

完成 code-spec 更新前：

- [ ] 内容具体且可操作？
- [ ] 包含了代码示例？
- [ ] 解释了为什么，而不仅是什么？
- [ ] 包含了可执行签名/契约？
- [ ] 包含了验证和错误矩阵？
- [ ] 包含了好/基本/坏案例？
- [ ] 包含了所需测试及断言点？
- [ ] 在正确的 code-spec 文件中？
- [ ] 没有重复现有内容？
- [ ] 新团队成员能理解吗？

## 与其他命令的关系

```
开发流程：
  学到东西 → `update-spec`（Trellis 命令）→ 知识已沉淀
       ↑                                  ↓
  `break-loop`（Trellis 命令）←──────────────────── 未来会话受益
  （深度 bug 分析）
```

- ``break-loop`（Trellis 命令）` - 深度分析 bug，通常会揭示需要更新的 spec
- ``update-spec`（Trellis 命令）` - 实际执行更新
- ``finish-work`（Trellis 命令）` - 提醒你检查是否需要更新 spec

## 核心理念

> **Code-spec 是活文档。每次调试会话、每个"顿悟时刻"都是让实现契约更清晰的机会。**

目标是**制度记忆**：
- 一个人学到的东西，所有人受益
- AI 在一个会话中学到的东西，延续到未来会话
- 错误变成文档化的护栏
