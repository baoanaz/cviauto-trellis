# 打破循环 — 深度 Bug 分析

当调试完成后，使用此技能进行深度分析，打破"修 bug → 遗忘 → 重复"的循环。

---

## 分析框架

从以下 5 个维度分析你刚修复的 bug：

### 1. 根因分类

此 bug 属于哪个类别？

| 类别 | 特征 | 示例 |
|----------|-----------------|---------|
| **A. 缺失 Spec** | 没有关于如何做的文档 | 新功能没有检查清单 |
| **B. 跨层契约** | 层间接口不清晰 | API 返回格式与预期不符 |
| **C. 变更传播失败** | 改了一处，漏了其他地方 | 改了函数签名，漏了调用点 |
| **D. 测试覆盖缺口** | 单元测试通过，集成测试失败 | 单独运行正常，组合后出问题 |
| **E. 隐式假设** | 代码依赖未文档化的假设 | 时间戳单位是秒还是毫秒 |

### 2. 修复失败原因（如适用）

如果你在成功之前尝试了多次修复，分析每次失败的原因：

- **表面修复**：修复了症状，而非根因
- **范围不完整**：找到了根因，但未覆盖所有情况
- **工具限制**：grep 漏掉了，类型检查不够严格
- **心智模型**：一直在同一层查找，未考虑跨层

### 3. 预防机制

什么机制可以防止此类问题再次发生？

| 类型 | 说明 | 示例 |
|------|-------------|---------|
| **文档** | 写下来让大家知道 | 更新思维指南 |
| **架构** | 从结构上使错误不可能发生 | 类型安全包装器 |
| **编译时** | 严格类型检查，无逃生通道 | 签名变更导致编译错误 |
| **运行时** | 监控、告警、扫描 | 检测孤儿实体 |
| **测试覆盖** | E2E 测试、集成测试 | 验证完整流程 |
| **代码审查** | 检查清单、PR 模板 | "你检查了 X 吗？" |

### 4. 系统性扩展

此 bug 揭示了哪些更广泛的问题？

- **类似问题**：其他地方可能还存在哪些同样的问题？
- **设计缺陷**：是否存在根本性的架构问题？
- **流程缺陷**：是否存在开发流程改进空间？
- **知识缺口**：团队是否缺少某些理解？

### 5. 知识沉淀

将洞察固化到系统中：

- [ ] 更新 `.trellis/spec/guides/` 思维指南
- [ ] 更新相关 `.trellis/spec/` 文档
- [ ] 创建问题记录（如适用）
- [ ] 为根因修复创建 feature ticket
- [ ] 如需要，更新检查指南

## 核心理念

> **调试的价值不在于修复 bug，而在于让这类 bug 永远不会再发生。**

三层洞察：
1. **战术层**：如何修复这个 bug
2. **战略层**：如何防止这类 bug
3. **哲学层**：如何扩展思维模式

30 分钟的分析可以节省 30 小时的未来调试时间。

## 思维框架：贝叶斯推理

当多个根因都有可能且证据不完整时，根据新证据按比例更新信念，而非固守初始假设。

### 步骤 1：建立先验

在调查之前，陈述你的信念及其依据：

| 假设 | 先验概率 | 推理依据 |
|------------|-------|-----------|
| H1：[原因 A] | 40% | 此模式最常见 |
| H2：[原因 B] | 30% | 在当前环境下合理 |
| H3：[其他] | 30% | 兜底 |

先验概率之和必须为 100%。如果无法分配概率，则先进行调查。

### 步骤 2：观察证据

记录发现了什么——要具体说明可靠性：

- 你到底观察到了什么？
- 可靠性如何？（测试输出 > 日志消息 > 用户报告 > 直觉）
- 多个假设是否都能解释此现象？

### 步骤 3：更新信念

对每个假设，问：**如果此假设为真，观察到这些证据的可能性有多大？**

更新方向比计算更重要：
- 证据被 H1 强烈预测 → H1 概率增加
- 证据与 H2 矛盾 → H2 概率降低
- 证据在所有假设下同等可能 → 不更新

### 步骤 4：寻找区分性证据

不要收集更多同类证据。找到在顶级假设之间**差异显著**的证据。

> 如果 H1 和 H3 接近："如果 H1 为真但 H3 为假，我会看到什么？"然后去找那个。

### 步骤 5：陈述置信度

| 置信度 | 行动 |
|------------|--------|
| 90%+ | 继续修复，监控 |
| 70-90% | 继续，添加回退检查 |
| 50-70% | 提交前先验证假设 |
| <50% | 需要更多证据，不要猜测 |

永远不要在证据不完整时表达二元的确定性。使用"最可能"、"可能但不太可能"、"值得调查"。

### 常见谬误

| 谬误 | 示例 | 纠正 |
|---------|---------|------------|
| **基础比率忽视** | "测试失败 → 代码有问题" | 测试因其他原因失败的频率有多高？ |
| **确认偏误** | "肯定是竞态条件，让我找竞态的证据" | 主动寻找反对你顶级假设的证据 |
| **锚定效应** | "上次是缓存问题，这次大概也是缓存" | 从当前上下文建立先验，而非昨天的 bug |

## 分析之后：立即行动

**重要**：完成上述分析后，你必须立即：

1. **更新 spec/guides** — 不要只列出 TODO，实际更新相关文件：
   - 如果是跨平台问题 → 更新 `cross-platform-thinking-guide.md`
   - 如果是跨层问题 → 更新 `cross-layer-thinking-guide.md`
   - 如果是代码复用问题 → 更新 `code-reuse-thinking-guide.md`
   - 如果是领域相关 → 更新 `backend/*.md` 或 `frontend/*.md`

2. **同步模板** — 更新 `.trellis/spec/` 后，同步到 `src/templates/markdown/spec/`

3. **提交 spec 更新** — 这是主要产出，不仅仅是分析文本

> **分析如果只留在聊天中就没价值。价值在于更新后的 spec。**
